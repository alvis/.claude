      Deviation 1 of 4

      Legacy circle geofences would have been silently skipped.

      A polygon-only breach test evaluates circle geofences as never-breached — so a subset of fleets would get no alerts at all, with nothing in the logs to say why. services/telemetry/src/geofence/breach.ts:106-149

      What the plan said

      Every geofence is a polygon. A ray-casting point-in-polygon test decides inside/outside for each ping.

      What the code revealed

      geofences.shape is a union. ~8%invented of rows are circles (center + radius_m) drawn by the old mobile tool before polygon editing existed. Ray-casting evaluates a circle as never-inside.

      The conservative choice

      Branch on shape: circles use haversine distance against radius_m; polygons keep ray-casting. Every active geofence is evaluated — none silently skipped.

      Revisit before merge?

      Circles could instead be normalized into 24-point polygons at read time, collapsing back to one code path — slightly less exact within a metre or two of the radius.

      How should the two-shape handling settle?

      Deviation 2 of 4

      The planned in-memory debounce dedupes nothing across replicas.

      This one changed the alert path's shape: de-duplication had to become shared state, which pulled Redis onto a path the plan treated as pure compute. services/telemetry/src/geofence/debounce.ts:1-63

      What the plan said

      Debounce repeat breaches with a 30-second in-memory window per (vehicle, geofence), so a vehicle hovering on an edge does not re-alert.

      What the code revealed

      The ingest consumer runs as many stateless replicas. Consecutive pings for one vehicle land on different replicas, so an in-process window dedupes nothing — a two-replica integration run re-alerted on every ping.

      The conservative choice

      Moved the window into Redis, keyed by (vehicle, geofence, edge) with a 30s TTL and SET NX. The ingest path already uses Redis for consumer offsets, so no new infrastructure — but the alert path now depends on it.

      Revisit before merge?

      As shipped, the path fails closed: if Redis is unreachable the breach is suppressed rather than risk a duplicate alert. That trades a missed alert for a duplicate — a policy call.

      Deviation 3 of 4

      Device clocks drift, so ordering by device time inverts breaches.

      Enter/exit ordering is load-bearing — get it wrong and a vehicle reads as outside a fence it is sitting inside. Device timestamps turned out to be untrustworthy. services/telemetry/src/geofence/breach.ts:106-149

      What the plan said

      Order breaches by the ping's device recorded_at so enter/exit pairs sequence correctly.

      What the code revealed

      Sampled ingest shows clocks drifting hours off, with a handful reporting year 2038invented. Ordering by recorded_at reorders and sometimes inverts enter/exit pairs.

      The conservative choice

      Order by server received_at; keep recorded_at as display-only metadata. Pings whose device time is >24h from server time go to ping_quarantine rather than being trusted — nothing dropped, sequence stays monotonic.

      Revisit before merge?

      The 24h bound is a guess. Tighter catches more bad clocks but risks quarantining legitimately delayed pings — vehicles out of coverage buffer offline and batch-upload on reconnect.

      How should the plausibility bound settle?

      Deviation 4 of 4

      A vehicle jittering on a geofence edge is an SMS storm.

      The only deviation with a dollar figure attached: unbounded breach alerts hit a billable SMS path. A cap now stands between GPS jitter and the invoice. services/telemetry/src/geofence/breach.ts:182-210

      What the plan said

      Each breach sends one alert via sendAlert(fleetId, breach). No rate handling mentioned.

      What the code revealed

      sendAlert() fans out to SMS with no per-fleet ceiling. A vehicle parked on an edge with GPS jitter fires rapid enter/exit breaches — hundreds of texts an hour for one vehicleinvented, each billable.

      The conservative choice

      Added a per-fleet hourly alert budget; past the cap, breaches roll into one "N more suppressed" digest, mirroring the existing email digest cap in notify.
