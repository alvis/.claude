      Domain explainer · migration mechanics

      Change the track while the train keeps moving.

      A live schema migration is safest when old and new application versions can coexist. The work is less like a switch and more like a sequence of reversible hand-offs.

      The idea

      Compatibility creates room to move.

      Do not ask every server and every record to change at the same instant. First make the system tolerate both shapes, then move data and traffic, and only then remove the old path. The plan is four moves — expand, dual write, backfill, then contract — each defined in the vocabulary below; hover one to light its entry.

      The invariant

      At every checkpoint, the currently deployed application must be able to read every record it can encounter and safely write a shape understood by the versions still receiving traffic.

      Four stages

      Expand, migrate, switch, contract.

      1. Expand the schema

        Add the new column or table without removing the old shape. Deploy code that can understand both, while continuing to use the established path.

        Rollback: remove unused addition db/migrate/20260714_expand_orders.rb:1-14

      2. Write both shapes

        New writes populate old and new locations. Observe mismatches before any reader depends exclusively on the new data.

        Pause: keep old reads authoritative app/models/order.rb:88-104

      3. Backfill and verify

        Move historical records in bounded batches. Compare counts, sampled values, and error rates until the new representation is complete enough to receive reads.

        Pause: resume from last batch app/jobs/backfill_orders.rb:1-46

      4. Switch, then contract

        Move reads gradually. After no supported version uses the old path, stop legacy writes and remove the obsolete schema in a separate deployment.

        Rollback: restore old reader db/migrate/20260901_contract_orders.rb:1-9

      What the simulation teaches

      Cohort size changes blast radius, not correctness. When backfill is incomplete, some new reads land on rows that only the old shape can answer — the fallback turns those from failures into quiet legacy reads. The authority switch is safe only when that amber band reaches zero and the rollback path has been exercised.

      Where it breaks

      Three boundaries deserve explicit proof.

      Open one boundary at a time — each names the failure, the proof it demands, and where that proof lives.

      Mixed application versions

      A rolling deploy means old and new binaries overlap. The schema must serve both until the final old instance is gone. observed

      Proof: the compatibility adapter reads either shape. app/adapters/order_compat.rb:12-40

      Partial backfill

      Readers need a defined fallback when a record has not migrated, rather than assuming the new location is complete. assumed

      This is the amber band in the rehearsal above: backfill gaps become quiet legacy reads only while the fallback stays on.

      Silent divergence

      Dual writes can disagree. Measure and reconcile the difference before changing which representation is authoritative. inferred

      Proof: a mismatch counter compares both shapes on every write. app/models/order.rb:96-104

      The irreversible moment comes last

      Dropping the old column or table is the only step that removes an easy rollback. Schedule it after telemetry shows no supported code path reads or writes the legacy shape.

      Walk the failing request path
      API write → compatibility adapter → legacy column + new column → mismatch telemetry → rollback threshold. Expand this before implementation to verify every owner and fallback.

      Before you implement

      Questions this plan tends to raise.

      The answers below are the ones reviewers ask first. Each carries the weight of its claim, so an assumption is never mistaken for a measurement.

      Can three application versions overlap during a rolling deploy?
      Yes. The expand step keeps every deployed build able to read and write both shapes, so a deploy that briefly runs three versions is safe — as long as none has entered the contract step. observed
      How long should dual write run before the read switch?
      Not a fixed duration. Keep dual writing until backfill is verified complete and the mismatch counter has stayed flat across a full traffic cycle. Residual mismatch is a blocker, not noise. inferred
      What single signal should trigger rollback?
      The new-read error budget or the mismatch rate crossing its configured threshold. Wire that trigger before the switch — a rollback you have not rehearsed is not a rollback. decided config/rollout.yml:3-9
      Is dropping the old column reversible?
      No. Contract is the one irreversible step. Schedule it in its own deploy, only after telemetry shows no supported code path reads or writes the legacy shape. observed
