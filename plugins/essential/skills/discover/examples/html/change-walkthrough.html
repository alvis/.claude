      1 / 4

      1 · The problem

      One flapping node paged the whole rotation.

      When a node oscillates in and out of a threshold, the check re-fires every evaluation interval. Before this change each fire opened its own incident and sent its own page, so a single flapping disk filled the on-call feed with dozens of identical incidents in minutes.

      The fix has to collapse the noise without hiding a genuine change of state.

      2 · The mechanism

      Collapse by origin with a fingerprint.

      Each alert now hashes its source and check_id into a fingerprint. If an open incident already carries that fingerprint, the alert folds into it — bumping a flap counter and refreshing last_seen — instead of opening a new one.

      Severity and message are deliberately left out of the key, so a check that wobbles between levels is still one incident.

      3 · The page gate

      A cooldown decides what actually pages.

      Folding an alert into an incident updates it, but the notifier only pages again once the five-minute cooldown has elapsed — unless the new severity is strictly higher than the severity that last paged, which bypasses the cooldown so a real escalation still wakes someone.

      This is the line that turns a noise fix into an on-call safety decision.

      4 · The edge that bites

      Dedup is scoped to open incidents only.

      The unique index is partial — WHERE state = 'open' — so once an incident is resolved its fingerprint is free again. A re-fire after resolution opens a fresh incident on purpose, which is what you want for a recurring-but-resolved condition, and surprising if you expected dedup to be permanent.

      Keep this in mind while reading the migration and answering the checks below.
