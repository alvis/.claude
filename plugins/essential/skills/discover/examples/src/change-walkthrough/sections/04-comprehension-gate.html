A disk check flaps warning → critical → warning within a minute. How many incidents exist, and what severity does the incident show?
Reveal the answer and why it matters

One incident, showing warning. The fingerprint excludes severity, so all three fires collapse into one open incident; inc.Severity is then overwritten by the latest alert, which was a warning. The incident can read lower than the worst level it hit — the reason the dedupe note flags peak-vs-current. services/alerts/dedupe.go:34

A critical pages and is acknowledged. Two minutes later the same check fires critical again — same level. Does it page?
Reveal the answer and why it matters

No — it updates silently. The severity is not strictly higher than the last paged severity, and two minutes is inside the five-minute cooldown, so shouldPage returns false. The incident's flap count and timestamps still advance; only the page is withheld. Acknowledgement does not close the incident. services/alerts/notifier.go:61

An incident is resolved (state = 'closed'). The same check fires again ten seconds later. What happens?
Reveal the answer and why it matters

It opens a brand-new incident. The unique index is partial — it only constrains rows WHERE state = 'open' — so a closed incident's fingerprint no longer blocks a new one. Recurring-but-resolved conditions get a fresh incident on purpose; do not expect dedup to be permanent. db/migrations/0087_incident_fingerprint.sql:12
