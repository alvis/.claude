With severity out of the key, a warning and a critical from the same check share one incident, and inc.Severity is overwritten to the latest alert — so an open incident can appear to downgrade from critical to warning. Confirm the incident view surfaces peak severity, not just current, or paging will read the right level while the UI reads the wrong one.

The bypass compares against NotifiedSeverity — the level at the last page — not the incident's current severity. A critical that de-escalates to warning and re-escalates to critical inside the window is treated as equal, so the second critical is suppressed. That is correct for flap and wrong for a genuine re-escalation; decide which one you are optimizing for.

medium services/alerts/notifier.go:61

Cooldown is measured from LastPagedAt, and the bypass is per-escalation, so a burst that ramps warning→critical still pages twice inside five minutes — once per level. That is intentional here, but it means the cooldown is not a hard once-per-window cap.

The partial WHERE state = 'open' is what scopes dedup to open incidents — a re-fire after resolution opens a fresh one by design. But the backfill UPDATE runs before the index is built, so if two already-open incidents share source+check_id the CREATE UNIQUE INDEX fails mid-migration. Check for colliding open incidents first, or fold them before this ships.
