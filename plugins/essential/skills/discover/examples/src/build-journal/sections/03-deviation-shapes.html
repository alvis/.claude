Legacy circle geofences would have been silently skipped.

A polygon-only breach test evaluates circle geofences as never-breached — so a subset of fleets would get no alerts at all, with nothing in the logs to say why. services/telemetry/src/geofence/breach.ts:106-149

What the plan said

Every geofence is a polygon. A ray-casting point-in-polygon test decides inside/outside for each ping.

What the code revealed

geofences.shape is a union. ~8%invented of rows are circles (center + radius_m) drawn by the old mobile tool before polygon editing existed. Ray-casting evaluates a circle as never-inside.

The conservative choice

Branch on shape: circles use haversine distance against radius_m; polygons keep ray-casting. Every active geofence is evaluated — none silently skipped.

Revisit before merge?

Circles could instead be normalized into 24-point polygons at read time, collapsing back to one code path — slightly less exact within a metre or two of the radius.
