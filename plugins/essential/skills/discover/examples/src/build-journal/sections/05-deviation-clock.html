Deviation 3 of 4

Device clocks drift, so ordering by device time inverts breaches.

Enter/exit ordering is load-bearing — get it wrong and a vehicle reads as outside a fence it is sitting inside. Device timestamps turned out to be untrustworthy. services/telemetry/src/geofence/breach.ts:106-149

What the plan said

Order breaches by the ping's device recorded_at so enter/exit pairs sequence correctly.

What the code revealed

Sampled ingest shows clocks drifting hours off, with a handful reporting year 2038invented. Ordering by recorded_at reorders and sometimes inverts enter/exit pairs.

The conservative choice

Order by server received_at; keep recorded_at as display-only metadata. Pings whose device time is >24h from server time go to ping_quarantine rather than being trusted — nothing dropped, sequence stays monotonic.

Revisit before merge?

The 24h bound is a guess. Tighter catches more bad clocks but risks quarantining legitimately delayed pings — vehicles out of coverage buffer offline and batch-upload on reconnect.
